Prompt Bank Database - How We Built It

Complete database-focused PDF content (easy + technical mixed).

1) Database Approach We Selected

We used a local JSON-based database model through PromptTyper local bridge API.

Main data file: prompt_bank_data.json

Reason: fast local setup, no cloud dependency, easy integration with existing Python tool.

2) Why We Chose Local JSON DB First

  • Project is local laptop based and PromptTyper already runs locally.
  • No user login requirement in current version.
  • Offline support needed.
  • Implementation speed high (good for rapid product iteration).

4) Core Prompt Schema

Each prompt row/object follows this structure:

{
  "id": "app-feature-spec",
  "title": "App Feature Spec Prompt",
  "category": "App Development",
  "tagline": "Use this prompt to define one app feature...",
  "prompt": "Act as a senior product manager...",
  "images": ["https://.../image.jpg" or "data:image/...base64,..."],
  "mainImage": "first image",
  "featuredInSlider": true
}

Validation rule: title and prompt mandatory; image optional.

5) How Data Is Normalized

Function: normalizePrompt(item, index)

  • Creates safe ID from title if ID missing.
  • Ensures category default is General.
  • Builds images[] list and ensures mainImage exists in list.
  • Standardizes featuredInSlider boolean.

This step guarantees DB consistency even if input payload varies.

7) Load and Save Flow (Step-by-Step)

Load Flow

  1. UI starts -> calls loadPromptLibrary().
  2. GET /prompt-bank-data from bridge.
  3. Payload prompts -> normalizePrompt().
  4. If API fails, fallback seeds are loaded.

Save Flow

  1. User adds/edits/deletes prompt in admin form.
  2. UI updates in-memory promptData.
  3. POST /prompt-bank-data with full prompt array.
  4. Bridge persists JSON to local DB file.

8) Image Storage Strategy

Project supports two image forms:

  • File path / URL (lightweight)
  • Base64 data URL (self-contained but heavy size)

Because of base64 image entries, DB file size can become large (for example 1GB risk if many heavy images).

Recommendation: keep DB text-focused, and store images in separate folder paths (or object storage in future).

9) Meta Data Model (Client-side)

In browser localStorage, key: prompt_bank_meta_v1

{
  "prompt-id-1": { "pinned": true, "favorite": false, "lastOpenedAt": 171... },
  "prompt-id-2": { "pinned": false, "favorite": true, "lastOpenedAt": 171... }
}

This is separated from main DB to keep business data clean and user preference flexible.

10) Error Handling in DB Layer

  • fetchWithTimeout() avoids hanging API calls.
  • Health check updates UI status: active / app not running.
  • If save fails, user sees explicit failure status instead of silent break.
  • Fallback data prevents blank UI when bridge unavailable.

11) What We Built Well in Database Design

  • Simple and maintainable schema.
  • Clear separation: prompt content vs user UI preferences.
  • Bridge-based access control (UI never writes random files directly).
  • Supports both text-only and image+text records.

12) Current Limitations

  • No relational indexing (JSON scan-based filtering).
  • No multi-user concurrency controls.
  • Large base64 images can inflate DB file size.
  • No version history/audit log yet.

13) Next Upgrade Path (If We Move to Real DB)

Option A - SQLite (Local, lightweight, best next step)

  • Tables: prompts, prompt_images, prompt_meta, activity_log.
  • Fast query + smaller storage + better integrity.

Option B - PostgreSQL/MySQL (Team/Cloud scale)

  • Multi-user, auth, backup, role-based management.

Image Best Practice

  • Store image file paths in DB, not raw base64 blob for large scale.

14) Quick Viva Answers (Database Specific)

Q: Is this a real DB?
A: Yes, current implementation is a local JSON document database pattern through API bridge.

Q: Why not SQL first?
A: Project priority was local speed and integration with existing Python software. JSON gave faster deployment.

Q: How do you ensure record consistency?
A: Normalization function standardizes fields before render/save.

Q: How can it scale?
A: Move same API contract to SQLite/Postgres with minimal UI changes.

15) Final Database Conclusion

Prompt Bank database was built as a practical local-first system: reliable for current single-user workflow, simple to operate, and ready for structured upgrade to SQL when scale increases.
